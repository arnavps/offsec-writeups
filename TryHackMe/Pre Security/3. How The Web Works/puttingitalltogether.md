# Putting It All Together

## Overview

This room ties together everything covered in the "How the Web Works" section — DNS, HTTP, and website fundamentals — and adds the infrastructure layer: load balancers, CDNs, databases, WAFs, and web servers. Understanding the full request lifecycle and the components involved is essential for mapping attack surface and understanding how web applications are deployed in production.

## Key Concepts

### The Full Web Request Lifecycle

When you visit a website, the following happens in sequence:

1. Browser checks its local DNS cache for the domain's IP
2. If not cached, a DNS query resolves the domain to an IP address
3. Browser sends an HTTP/HTTPS request to that IP
4. The request may pass through a WAF and/or load balancer before reaching the web server
5. The web server processes the request, potentially querying a database
6. The server returns an HTTP response containing HTML, CSS, JS, images, etc.
7. The browser renders the content

---

### Load Balancers

A load balancer sits in front of multiple web servers and distributes incoming traffic across them. Used when a single server can't handle the volume of traffic, or when high availability is required.

**Key functions:**
- Distributes requests across multiple backend servers
- Performs health checks — if a server goes down, the load balancer stops sending traffic to it (failover)

**Common algorithms:**
- **Round-robin** — sends each request to the next server in rotation
- **Weighted** — sends requests to the least busy server based on current load

---

### CDN (Content Delivery Network)

A CDN hosts static assets (JavaScript, CSS, images, videos) across servers distributed globally. When a user requests a static file, the CDN serves it from the nearest geographic location rather than the origin server.

Benefits:
- Reduces latency for end users
- Offloads traffic from the origin web server
- Improves availability and resilience

---

### Databases

Web applications store and retrieve data using databases. The web server communicates with the database to read/write user data, content, configurations, etc.

Common databases:

| Database | Type |
|---|---|
| MySQL | Relational (SQL) |
| MSSQL | Relational (SQL) — Microsoft |
| PostgreSQL | Relational (SQL) |
| MongoDB | Non-relational (NoSQL) |

The type of database in use is relevant for SQL injection testing — different databases have different syntax and features.

---

### WAF (Web Application Firewall)

A WAF sits between the client and the web server, inspecting HTTP requests for malicious patterns before they reach the application.

**What a WAF does:**
- Detects and blocks common attack patterns (SQLi, XSS, path traversal, etc.)
- Verifies requests come from real browsers rather than automated bots
- Enforces rate limiting — blocks IPs sending too many requests per second (mitigates brute force and DDoS)
- Drops suspicious requests before they reach the web server

WAFs are a defensive layer, not a complete solution — they can be bypassed with obfuscation and encoding techniques.

---

### Web Servers

A web server is software that listens for incoming HTTP connections and serves content to clients.

**Common web servers:**

| Software | Default Root Directory |
|---|---|
| Apache | `/var/www/html` (Linux) |
| Nginx | `/var/www/html` (Linux) |
| IIS | `C:\inetpub\wwwroot` (Windows) |
| Node.js | Defined in application code |

**Virtual Hosting:**
A single web server can host multiple websites with different domain names. It uses the `Host` header from the HTTP request to determine which site to serve. Each site is defined in a text-based virtual host configuration file. If no matching virtual host is found, the server returns the default site.

---

### Static vs Dynamic Content

| Type | Description | Examples |
|---|---|---|
| Static | Fixed content served directly from disk — never changes per request | Images, CSS, JS files, static HTML |
| Dynamic | Content generated at request time based on logic, user data, or database queries | Blog feeds, search results, user dashboards |

Dynamic content is generated by backend scripting languages running on the server. The user only sees the output (HTML) — the backend code is never exposed.

---

### Backend Scripting Languages

Backend languages run on the server and generate dynamic responses. They can interact with databases, process user input, call external APIs, and more.

Common backend languages:

- PHP
- Python
- Ruby
- Node.js (JavaScript server-side)
- Perl

**Example — PHP dynamic content:**

```
http://example.com/index.php?name=adam
```

The server-side PHP script reads the `name` parameter and generates a personalised response. The user never sees the PHP code — only the resulting HTML.

## Important Terminology

| Term | Definition |
|---|---|
| Load Balancer | Distributes traffic across multiple servers for performance and availability |
| Failover | Automatically redirecting traffic away from a failed server |
| CDN | Content Delivery Network — serves static assets from geographically distributed servers |
| Database | Stores and retrieves application data |
| WAF | Web Application Firewall — filters malicious HTTP requests |
| Rate Limiting | Restricting the number of requests from a single IP within a time window |
| Web Server | Software that serves HTTP content (Apache, Nginx, IIS) |
| Virtual Host | A configuration allowing one web server to serve multiple domains |
| Static Content | Fixed files served directly without server-side processing |
| Dynamic Content | Content generated at request time by backend code |
| Backend Language | Server-side scripting language (PHP, Python, Node.js, etc.) |

## Workflow / Process

### Full Request Lifecycle

```
User enters URL
        |
DNS resolution → IP address obtained
        |
HTTP request sent to IP
        |
WAF inspects request → block or allow
        |
Load balancer routes to available server
        |
Web server receives request
        |
Backend code executes (if dynamic) → queries database if needed
        |
Response generated → HTML/CSS/JS returned
        |
CDN may serve cached static assets
        |
Browser renders the page
```

## Real-World Relevance

- Load balancers can complicate testing — responses may differ between backend servers, making consistent exploitation harder
- CDNs (especially Cloudflare) mask the real origin server IP — bypassing the CDN to find the origin is a common recon step
- Identifying the database type is critical for SQL injection — MySQL, MSSQL, and PostgreSQL each have different syntax and exploitation paths
- WAFs are a primary obstacle in web app pentesting — WAF bypass techniques (encoding, obfuscation, chunked requests) are a dedicated skill area
- Virtual host enumeration (`Host` header fuzzing) can reveal hidden internal sites on the same server
- Knowing the backend language helps predict vulnerability patterns — PHP has different common issues than Node.js or Python

---

## Key Learnings

- A web request passes through multiple infrastructure layers before reaching the application
- Load balancers provide scalability and failover; CDNs reduce latency for static assets
- WAFs filter malicious requests but can be bypassed — they are a layer of defence, not a guarantee
- Web servers use virtual hosting to serve multiple domains from one machine
- Dynamic content is generated server-side; users only see the output, never the code
- The backend language and database type directly influence what vulnerabilities are likely present

---

## Additional Notes

- Tools like `whatweb`, `wappalyzer`, and HTTP response headers can fingerprint the web server, backend language, and frameworks in use
- The `Server` and `X-Powered-By` response headers often reveal the web server software and backend language — useful for recon
- Origin IP discovery behind a CDN can be done via historical DNS records, SSL certificate transparency logs, or direct subdomain enumeration

---

## Conclusion

The web is not just a browser and a server — it's a layered infrastructure of DNS resolvers, load balancers, CDNs, WAFs, databases, and application code all working together. Understanding each component's role, and where it can be probed or exploited, gives a complete picture of the web application attack surface.
