# Introduction to SIEM

## Overview

A SIEM (Security Information and Event Management) system is the central platform in most SOC environments. It solves a fundamental problem in security operations: devices across a network generate enormous volumes of logs constantly, and no analyst team can manually read all of them across every machine. SIEM collects, normalises, and correlates all of that log data into one place, enabling real-time alerting and investigation. This room covers why SIEM exists, what problems it solves, how it works, and how logs from different sources are ingested into it.

## Key Concepts

### The Problem: Logs Everywhere, Answers Nowhere

Every device in a network — endpoints, servers, firewalls, routers — continuously generates logs of its own activity. These logs are invaluable for detecting attacks and investigating incidents, but working with them without a centralised solution creates serious problems:

| Challenge | Description |
|-----------|-------------|
| **Numerous log sources** | A single network can have dozens or hundreds of devices, each generating hundreds of events per second |
| **No centralisation** | Logs sit on the machines that generate them — analysts must SSH or RDP into each device individually to read them |
| **Limited context** | Individual logs from a single device often look harmless. Correlation across sources reveals the full story |
| **Limited analysis** | Manually reading logs from all devices to find anomalies is practically impossible at volume — critical events will be missed |
| **Format inconsistency** | Windows, Linux, firewalls, and web servers all generate logs in different formats — understanding each requires specialised knowledge |

#### Why Correlation Matters

A single file access event on a workstation looks normal in isolation. But correlate it with:
- A VPN login from an unfamiliar geographic IP
- A PowerShell script execution on the same host
- An outbound connection to an external IP shortly after

...and the picture changes completely. SIEM makes this cross-source correlation automated and continuous.

### Core SIEM Features

#### Centralised Log Collection

SIEM pulls logs from all sources across the network — endpoints, servers, firewalls, web applications — into a single platform. Analysts investigate everything from one interface rather than jumping between machines.

Logs are collected via lightweight agents installed on endpoints, or through network protocols like Syslog.

#### Log Normalisation

Raw logs from different devices arrive in different formats. A Windows event log looks nothing like an Apache web server log or a Cisco firewall log.

- **Parsing** — breaking down a raw log into its individual fields (timestamp, user, action, IP, etc.)
- **Normalisation** — converting logs from all sources into a single, consistent format

Normalisation allows analysts and detection rules to query all log sources using a unified field structure, regardless of where the log originated.

#### Log Correlation

SIEM applies logic to find relationships between events across multiple sources and timeframes. Detection rules define patterns of activity that, when matched, indicate a potential threat.

Example of correlated activity over 5 minutes:
1. User `haris` logs in via VPN from a previously unseen IP address
2. `haris` accesses documents on a shared drive
3. `haris` executes a PowerShell script
4. The system makes an outbound network connection to an external IP

Individually, each event appears routine. Correlated, they suggest credential compromise and potential data exfiltration.

#### Real-time Alerting

SIEM detection rules run continuously against incoming log data. When a rule's conditions are met, an alert fires and the analyst is notified. Rules can come built-in with the SIEM, or be custom-written by detection engineers and L2/L3 analysts to address specific threats relevant to the organisation.

#### Dashboards and Reporting

Dashboards provide a summarised, visual overview of the security environment. Common dashboard components include:

- Active alert highlights
- System and health notifications
- Failed login attempt counts
- Events ingested per timeframe
- Rules triggered
- Top domains visited
- Log source status

Each SIEM provides default dashboards and the ability to build custom views tailored to the organisation's needs.

### Log Source Categories

#### Host-Centric Log Sources

Events generated by or on an individual host.

- User accessing a file
- Authentication attempts (login success or failure)
- Process execution
- Registry key creation, modification, or deletion
- PowerShell script execution

Devices: Windows endpoints, Linux servers, application servers, databases.

#### Network-Centric Log Sources

Events generated by network communication between hosts or between hosts and the internet.

- SSH connection attempts
- File transfers via FTP
- Web traffic (HTTP/HTTPS requests)
- VPN access events
- Network file sharing activity

Devices: Firewalls, IDS/IPS, routers, switches, VPN gateways.

### Common Log Locations

#### Windows

Windows logs are viewable through the **Event Viewer** application. Events are assigned unique Event IDs for easy filtering. Logs from Windows endpoints are forwarded to the SIEM via an agent or Windows Event Forwarding (WEF).

#### Linux

| Path | Contents |
|------|---------|
| `/var/log/httpd` | HTTP request/response and Apache error logs |
| `/var/log/cron` | Cron job execution events |
| `/var/log/auth.log` or `/var/log/secure` | Authentication events (login attempts, sudo usage, SSH) |
| `/var/log/kern` | Kernel-level events |

#### Web Server

Apache web server logs are typically found at `/var/log/apache` or `/var/log/httpd`. These record all HTTP requests and responses — essential for detecting web-based attacks.

### Log Ingestion Methods

| Method | Description |
|--------|-------------|
| **Agent / Forwarder** | A lightweight software agent installed on the endpoint captures and forwards logs to the SIEM. Splunk calls this component a "Forwarder". Most common method for endpoint log collection |
| **Syslog** | A widely supported network protocol for sending log data in real time from devices (servers, network appliances, databases) to a centralised receiver. Many devices support Syslog natively |
| **Manual Upload** | Some SIEMs (Splunk, Elastic) allow offline log files to be uploaded directly for one-off analysis — useful for investigating an isolated incident |
| **Port Forwarding** | The SIEM listens on a specific port and devices are configured to forward their logs to that port. Common for network devices that can't run agents |

## Important Terminology

| Term | Meaning |
|------|---------|
| SIEM | Security Information and Event Management — centralised log collection, correlation, and alerting platform |
| Log source | Any device or application that generates log data |
| Parsing | Breaking down a raw log entry into structured fields |
| Normalisation | Converting logs from diverse formats into a consistent, unified structure |
| Correlation | Finding relationships between events from different sources to identify patterns |
| Detection rule | Logic configured in a SIEM that triggers an alert when matched against log data |
| Agent/Forwarder | Lightweight software installed on an endpoint to collect and forward logs |
| Syslog | Network protocol for real-time log transmission to a centralised receiver |
| UBA | User Behaviour Analytics — SIEM capability that baselines user activity and flags deviations |

## Real-World Relevance

- SIEM is the primary tool used daily by SOC analysts in enterprise environments — proficiency with at least one platform (Splunk, Microsoft Sentinel, Elastic, IBM QRadar) is a baseline expectation for L1/L2 SOC roles
- Log correlation is what separates a basic alert from a confirmed incident — SIEM is the tool that makes correlation at scale possible
- Attackers that operate in phases (initial access, then lateral movement, then data exfiltration) can only be caught if logs from multiple stages are correlated — SIEM enables this
- Detection engineering (writing custom SIEM rules) is a high-value skill — it directly determines what threats get caught
- Modern SIEMs include UEBA (User and Entity Behaviour Analytics) powered by machine learning — detecting anomalous user behaviour that no static rule would catch
- From an offensive perspective, understanding what SIEM logs and correlates informs evasion — attackers who understand detection rules can operate in ways designed to avoid triggering them

---

## Key Learnings

- SIEM solves the core problems of distributed, inconsistent, high-volume log data in a network
- The two categories of log sources are host-centric (endpoint activity) and network-centric (communication events)
- SIEM parses raw logs into fields and normalises them into a consistent format for unified querying
- Correlation across multiple log sources reveals threats that individual logs alone would not expose
- Detection rules trigger real-time alerts; dashboards provide operational visibility
- Logs are ingested via agents, Syslog, manual upload, or port forwarding depending on the source type

---

## Additional Notes

- The choice of SIEM platform matters — Splunk is the most widely used commercial option, Microsoft Sentinel is dominant in Azure environments, and Elastic SIEM is a popular open-source alternative
- A SIEM is only as good as its log sources — if a device is not sending logs to the SIEM, activity on that device is invisible to the SOC
- Log retention policies directly affect how far back an investigation can go — short retention windows are a significant limitation during forensic investigations
- Tuning detection rules to reduce false positives is a continuous, ongoing process — a noisy SIEM leads to alert fatigue and missed real threats

---

## Conclusion

SIEM is the operational backbone of modern security monitoring. It transforms the impossible task of manually reviewing logs from hundreds of devices into a manageable, automated process with real-time alerting and contextual correlation. Understanding how SIEM ingests, normalises, and correlates logs — and how detection rules drive alerts — is foundational knowledge for any SOC analyst, detection engineer, or security professional working in a defensive capacity.
